# Skia Backend Divergence Audit

Notes from a focused parallel audit comparing the experimental Skia backend (`src/mbgl/skia/`) against the established OpenGL backend (`src/mbgl/gl/`). Each entry is a concrete, code-evidenced divergence with the file/line and an estimate of why it bites at runtime. Findings are grouped by area.

The Skia backend at high pitch + high zoom drops to ~2 fps where OpenGL holds 120 Hz on the same hardware. The audit covers correctness drift (slashes, missing fills, ordering) and the 50–100× perf gap.

## Clipping And Ordering

1. **Tile clipping is per-drawable canvas-clip vs GL's per-tile stencil.** `src/mbgl/skia/layer_group.cpp:84` only applies the tile clip when `drawable.getEnableStencil()` is true. GL renders the stencil mask once per layer-group via `Context::renderTileClippingMasks` (`src/mbgl/gl/layer_group_gl.cpp:93`) and applies it uniformly. With overscan or wrap copies, drawables whose `enableStencil` flag is false leak into adjacent tile regions.
2. **Near-plane clipping is selective on Skia, universal on GL.** `clipProjectedTriangles` in `src/mbgl/skia/drawable.cpp:391` only runs for `rasterDrawable || solidFillDrawable || backgroundPatternDrawable`. Lines, line patterns, line gradients, line SDF, fill outlines, fill patterns, and fill extrusions render unclipped. GL's GPU near-plane test catches all of them in the rasterizer. This is the direct cause of the diagonal slashes seen at pitch=85 over Seattle (highway line drawables fan out across the canvas when their triangles span the near plane).
3. **`gfx::ColorMode` channel mask is binary on Skia.** `src/mbgl/skia/drawable.cpp:2628` early-returns when any channel write is disabled (`hasColorWrites` returns false on `mask.r||g||b||a` falsy). GL applies the mask per-channel via `glColorMask`. Any layer that wants to write some channels but not others gets nothing on Skia.
4. **Depth mode is silently degraded.** No equivalent to GL's `Context::setDepthMode`. Drawables with `enableDepth=true` (3D fill-extrusion most prominently) render in drawable order, not depth-sorted. Z-fighting with overlapping extrusions is the documented deferral class.
5. **Drawable order workaround.** `src/mbgl/skia/layer_group.cpp:107` reorders by `overscaledZ` ascending each frame as a stand-in for stencil masking. This kills the worst overscan bleeds but is fragile when wrap copies of the same logical tile exist at the same zoom.
6. **Lines via `canvas->drawLine` for Lines mode.** `src/mbgl/skia/drawable.cpp:4501` projects to screen space first, then calls `canvas->drawLine` per segment. This path is for `DrawModeType::Lines` (debug, collision boxes), not the main line layers — those go through `SkMesh` with `kTriangles`. Worth flagging because debug-only lines lose perspective correction, which doesn't affect the styled map but shows up in collision-debug overlays.

## Mesh Geometry And Vertex Attributes

7. **`MeshVertex` is one fat conflated struct.** `src/mbgl/skia/drawable.cpp:170` defines a single `MeshVertex` (~213 bytes of float fields) that every shader path reads from. Line meshes use packed attributes (`lineNormal`, `lineWidth`, `lineBlur`, `lineProgress`) that should not be linearly interpolated, but `interpolateVertex` (line 378) blits everything as floats. This is why extending `clipProjectedTriangles` to lines broke line rendering — interpolation between two clipped vertices corrupts the packed semantics.
8. **GL keeps line attributes packed and unpacks per-vertex in the shader.** `include/mbgl/shaders/gl/line.hpp:127` decodes `a_pos_normal` (one `vec2` of int16s) inside `main()` via `floor(a_pos_normal * 0.5)` and `a_pos_normal - 2*pos`. Skia unpacks once at mesh-build time and stores the unpacked floats in `MeshVertex`. After unpacking, linear interpolation between two adjacent unpacked vertices is meaningless for some fields (e.g., `lineProgress` blends correctly but two opposite normals interpolate to zero).
9. **Line mesh near-plane clipping needs to be either polyline-level or attribute-aware.** Two viable shapes: (a) clip the polyline before the line bucket emits triangles (renderer-level work, not Skia-specific), or (b) write a line-aware Sutherland–Hodgman that reconstructs canonical packed attributes before interpolating. Option (a) is the right architectural answer; option (b) is a Skia-local hack.

## Performance Hotspots

The 50–100× gap traces to the per-drawable per-frame work the Skia backend does that GL does once or on the GPU. Numbers are rough orders of magnitude per frame.

10. **Per-frame `MeshVertex` vector allocation and rebuild.** `src/mbgl/skia/drawable.cpp:4048` does `std::vector<MeshVertex> meshVertices(vertexReader.count)` every render call for every drawable. GL streams vertex buffers once during `upload()`. At high zoom this is O(total_vertices) of allocation + copy per frame, easily millions of bytes per frame for a busy map. The single biggest item.
11. **Per-vertex CPU lighting/color computation for fill-extrusion and symbol drawables.** `src/mbgl/skia/drawable.cpp:4386–4435` runs branching + dot products + `std::pow` per vertex per frame for fill-extrusion lighting; symbol drawables unpack and re-interpolate colors per vertex. GL computes these in the GPU vertex shader. Disproportionately expensive when fill-extrusion is active.
12. **CPU Sutherland–Hodgman for raster/solid-fill at high pitch.** `clipProjectedTriangles` (`src/mbgl/skia/drawable.cpp:391`) is O(triangles × edges) with a per-triangle vector allocation. `needsProjectedClipping` returns true for every raster tile at high pitch, so every visible tile pays the polygon-clipping tax. GL's hardware clipper does this in the rasterizer for free.
13. **Per-frame stable_sort of all drawables.** `src/mbgl/skia/layer_group.cpp:139` allocates a vector and sorts every layer's drawables every frame. At high zoom each layer can have 50–200 drawables; sort is O(N log N) per layer, allocation is O(N). GL has no equivalent because stencil makes order less important.
14. **Per-tile `SkPath::Polygon` reconstruction.** `tileClipPath` (`src/mbgl/skia/layer_group.cpp:41`) builds a fresh four-point `SkPath` per drawable per frame by projecting the tile corners. GL renders the stencil mask once per frame for the layer, not once per drawable.
15. **Per-drawable `image->makeShader(...)` calls.** Pattern fills, raster, line patterns — all wrap their `SkImage` into a fresh `SkShader` (and sometimes a `makeSubset` first) every frame around `src/mbgl/skia/drawable.cpp:4965+`. GL binds textures by handle; no shader-tree allocation per frame.
16. **Per-drawable uniform-blob encoding.** `writeUniform` is called 20–50 times per drawable per frame (`src/mbgl/skia/drawable.cpp:4658–4860` is dominated by these calls), with no caching even when the same drawable's uniforms haven't changed since last frame. GL uses `MLN_UBO_CONSOLIDATION` to batch uniform updates and share UBOs across drawables.
17. **Mesh specifications are correctly cached.** Worth noting because it's the one piece that's right: `static sk_sp<SkMeshSpecification>` patterns at `src/mbgl/skia/drawable.cpp:514+` reuse spec objects across frames. Vertex/index buffers and uniforms still rebuild per frame, though, so the cached spec is mostly a CPU-time win during shader compilation, not the per-frame hot path.

## Highest-Leverage Next Steps

- **Cache per-drawable mesh data across frames** (#10). The work in `Drawable::draw` to populate `meshVertices` is currently O(verts) per frame. Largest single perf win — but the cache cannot be gated on `VertexAttributeArray::isModifiedAfter()` alone: meshVertex content depends on per-frame uniforms (`color`/`colorT`/`opacityT` interp factors, `circleColor`, `fillExtrusionOpacity`, fill-extrusion light direction/color/etc., symbol fill/halo colors, symbol opacity, `vertexOpacity`). A correct cache must invalidate on uniform change too — either via a uniform-buffer fingerprint/hash or by limiting caching to drawable types whose meshVertex content is purely attribute-derived (a much smaller subset than expected). First attempt rolled back; needs a uniform-aware design before the next try.
- **Reuse the per-drawable scratch vectors** (a partial of #10). `meshVertices` and `clippedIndexes` are stack-local in every `draw` call. Hoisting them to mutable members on `skia::Drawable` and clearing/reserving instead of reallocating saves the heap traffic (the allocator alone is a significant slice at high vertex counts) without any correctness implications.
- **Polyline-level near-plane clipping in the line bucket** (#2 + #7 + #9). Fixes the slashes correctness bug *and* eliminates one motivation for line-level CPU clipping. Renderer-level change, shared with other backends, but the right architectural answer.
- **Consolidate uniform encoding via `MLN_UBO_CONSOLIDATION`** (#16). The infrastructure already exists in the renderer; the Skia backend mostly opts out. Wiring it up should remove dozens of `memcpy`s per drawable per frame.
- **Lift `image->makeShader` into the texture upload path** (#15). `Texture2D::upload` already produces an `SkImage`; the wrapping `SkShader` can be cached on the texture and invalidated on `setSamplerConfiguration` / `setSize` instead of rebuilt per drawable.
- **Reuse tile clip paths per frame** (#14). One projection + path build per tile per frame, not per drawable. Also opens the door to subtraction-based stencil emulation for the overscan/wrap-copy cases (#1).
